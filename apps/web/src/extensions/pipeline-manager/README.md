# Pipeline Manager Agent

The Pipeline Manager is the **data-ecosystem reliability specialist** in Lattik Studio. It monitors, troubleshoots, and operates the two surfaces that move data through the platform:

- **Logger Tables** — event ingestion through Kafka into Iceberg, written by the logger-writer pipeline.
- **Airflow DAGs** — materialization of Lattik Tables on top of Logger Tables (and on top of other Lattik Tables).

It picks up where the Data Architect leaves off: the architect defines what the data ecosystem looks like, the manager keeps it running.

## Architecture

```
User ←→ Chat ←→ Agent (Claude Sonnet 4.6) ←→ Tools ←→ Canvas / Airflow API / DB
```

The agent is a `ToolLoopAgent` (Vercel AI SDK v6) with a max of 10 tool steps per turn. It loads a skill markdown doc via `getSkill` before starting work — the skill is the source of truth for the workflow, not memory.

### Key Behavior
- **Skill-driven:** Every workflow starts with `getSkill`. Three DAG skills ship today: `monitoring-dags`, `triggering-runs`, `troubleshooting-failures`. Logger Table skills are the next deliverable.
- **Canvas-first (DAG side):** On any DAG monitoring or overview request, the first tool call after `getSkill` is `renderDagOverview`. Detail views go through `renderDagRunDetail`. The canvas IS the starting point — no clarifying questions before rendering.
- **Render-tool-only:** No `spec` code fences or raw JSONL patches. All canvas rendering goes through the dedicated render tools.
- **Lattik-only scope:** Only Lattik-managed surfaces are in scope — DAGs tagged `lattik` and Logger Tables registered through the platform. Unrelated DAGs or external data systems are out of scope.
- **Read + control, not edit:** The agent can trigger runs, retry tasks, pause/unpause schedules — but it does NOT modify DAG structure or Logger Table definitions. Definition changes go back through the Data Architect.

> **Status:** The agent's identity and prompt cover both Logger Tables and Airflow DAGs, but the **tools, skills, and canvas components for Logger Table monitoring are not yet implemented** — only the DAG side ships today. See [`docs/extensions/pipeline-manager.md`](../../../../../docs/extensions/pipeline-manager.md) for the planned shape.

## Tools

| Tool | Purpose |
|------|---------|
| `getSkill` | Load a skill markdown document by ID |
| `listDags` | Query Airflow REST API for Lattik-managed DAGs (filtered by `lattik` tag) |
| `getDagDetail` | Fetch a DAG's structure, schedule, and the linked Lattik Table spec |
| `listDagRuns` | Run history for a DAG (state, logical date, start/end time, duration) |
| `getTaskInstances` | Task instances for a specific run (state, duration, try number) |
| `getTaskLogs` | Stdout/stderr for a specific task instance (Spark driver output, sensor pokes) |
| `renderDagOverview` | Render the DAG list canvas with status badges and recent-run sparklines |
| `renderDagRunDetail` | Render the task graph + log viewer for a specific run |
| `readCanvasState` | Read user selections from the canvas (e.g. which DAG row was clicked) |
| `handback` | Return control from the agent (`pause` for off-topic, `complete` when done) |

The Airflow REST API client lives at [`lib/airflow-client.ts`](lib/airflow-client.ts). All read-side tools (`listDags`, `getDagDetail`, `listDagRuns`, `getTaskInstances`, `getTaskLogs`) go through it.

> **Note:** Trigger / backfill / retry / pause-unpause tools are part of the design (see [`docs/extensions/pipeline-manager.md`](../../../../../docs/extensions/pipeline-manager.md)) but are not yet implemented — only the read-side tools ship today.

## Canvas

The canvas uses `@json-render/react`. Two main spec layouts:

- **Overview:** DAG list with status badges, last-run outcome, recent-run sparkline. Generated by `renderDagOverview`.
- **Run detail:** Task graph for one run, with each task showing state and duration. Selecting a task surfaces its logs. Generated by `renderDagRunDetail`.

State binding is one-way for now (agent → canvas); user interaction comes back via `readCanvasState`.

```
│   ├── index.ts                     Skill metadata and loader

│   ├── triggering-runs.md           Manual trigger + backfill workflow
│   └── troubleshooting-failures.md  Diagnostic playbook (sensor timeout, OOM, S3 access, etc.)
└── tools/
    ├── get-skill.ts             Load skill documents
    ├── list-dags.ts             Discover Lattik-tagged DAGs
    ├── get-dag-detail.ts        Inspect a DAG's structure + linked spec
    ├── list-dag-runs.ts         Run history
    ├── get-task-instances.ts    Task-instance state for a run
    ├── get-task-logs.ts         Fetch task logs
    ├── render-dag-overview.ts   Render DAG list canvas
    ├── render-dag-run-detail.ts Render task graph + logs canvas
    └── read-canvas-state.ts     Read current canvas state
```
