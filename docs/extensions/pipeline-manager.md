# Pipeline Manager Agent

Design doc for the Pipeline Manager specialist agent — the data-ecosystem reliability specialist. It owns the operational lifecycle of the two surfaces that move data through the platform:

- **Logger Tables** — event ingestion through Kafka into Iceberg, written by the logger-writer pipeline.
- **Airflow DAGs** — materialization of Lattik Tables on top of Logger Tables (and on top of other Lattik Tables), generated by `dag-generator.ts` after a Data Architect PR merges.

## Scope

The Data Architect agent handles **defining** the data ecosystem (entities, dimensions, Logger Tables, Lattik Tables, metrics) and submitting PRs. The Pipeline Manager picks up after merge — it handles **monitoring, troubleshooting, triggering, and controlling** both Logger Tables (ingestion health) and Airflow DAGs (materialization runs).

It does NOT modify Logger Table or DAG definitions. To change a pipeline's definition, the user goes back to the Data Architect.

> **Implementation status:** The DAG operator capabilities below are designed; Phase 1 (read-only DAG monitoring) ships today. Logger Table monitoring is part of the agent's identity and remit but its tools, skills, and canvas components are the next deliverable — see [Logger Table monitoring (planned)](#logger-table-monitoring-planned) below.

## Architecture

```
User ←→ Chat ←→ Pipeline Manager Agent (Claude Sonnet 4.6) ←→ Tools ←→ Canvas / Airflow API / DB / S3
```

Same pattern as Data Architect: `ToolLoopAgent` with skills, canvas rendering via `@json-render/react`, and the handoff protocol (`handback` with `pause`/`complete`).

## Capabilities

| Capability | Description |
|---|---|
| **List DAGs** | Show all Lattik-managed DAGs with schedule, status (active/paused), and last run outcome |
| **DAG detail** | View a DAG's task graph, schedule, config, and the underlying Lattik Table definition it was generated from |
| **Trigger run** | Manually trigger a forward-run DAG for a specific `ds` (logical date) |
| **Trigger backfill** | Kick off backfill DAGs with a date range, respecting parallelism settings from the table spec |
| **Monitor runs** | Show DAG run history, task instance states (success/running/failed/up_for_retry), durations |
| **View task logs** | Fetch logs for a specific task instance (e.g. why did a Spark job fail?) |
| **Retry failed tasks** | Clear and retry individual failed task instances |
| **Pause / Unpause** | Toggle a DAG's scheduled execution on/off |
| **Troubleshoot** | Guided troubleshooting — correlate failures with Spark driver logs, sensor timeouts, OOM errors |

## Tools

| Tool | Type | Description |
|---|---|---|
| `getSkill` | stateless | Load skill markdown doc by ID |
| `listDags` | stateless | Query Airflow REST API for Lattik-managed DAGs (filtered by `lattik` tag), joined with definitions DB for context |
| `getDagDetail` | stateless | Fetch DAG structure, config, schedule, and linked Lattik Table spec from definitions DB |
| `listDagRuns` | stateless | Get run history for a DAG (state, logical date, start/end time, duration) |
| `getTaskInstances` | stateless | Get task instances for a specific DAG run (state, duration, try number) |
| `getTaskLogs` | stateless | Fetch stdout/stderr logs for a task instance (Spark driver output, sensor poke output) |
| `triggerDagRun` | stateless | POST to Airflow API to trigger a manual run with `logical_date` conf |
| `triggerBackfill` | stateless | Trigger backfill DAG with start/end date range parameters |
| `retryTask` | stateless | Clear a failed task instance state so it retries |
| `toggleDagPause` | stateless | Pause or unpause a DAG's scheduled execution |
| `renderDagOverview` | stateless | Render canvas with DAG list, status badges, sparkline of recent runs |
| `renderDagRunDetail` | stateless | Render canvas with task graph, instance states, log viewer |
| `readCanvasState` | factory | Read current canvas state (user selections in the overview) |
| `handback` | inline | Pause/complete handoff protocol (same as data-architect) |

### Tool patterns

Follows the same patterns as the Data Architect extension:

- **Stateless tools** are plain objects with `description`, `inputSchema` (via `zodSchema()`), and `execute`
- **Factory tools** (`readCanvasState`) receive `getCanvasState` closure at agent creation time
- **`handback`** is defined inline in `agent.ts`
- All zod fields include `.describe()` for the LLM
- Spec-returning tools (render*) return `{ kind, spec, instruction }` where `instruction` tells the agent what NOT to do (e.g. don't repeat the data in chat)
- Mutating tools (`triggerDagRun`, `retryTask`, `toggleDagPause`) return confirmation objects with the action taken

## Skills

| Skill ID | Title | Description |
|---|---|---|
| `monitoring-dags` | Monitoring DAG Health | Workflow for checking DAG status — list DAGs, drill into failures, view logs, retry. Covers the read-only monitoring loop. |
| `triggering-runs` | Triggering Runs & Backfills | Workflow for manual triggers and backfills — select DAG, choose dates, trigger, monitor progress. Covers both forward-run and backfill DAGs. |
| `troubleshooting-failures` | Troubleshooting Failures | Guided troubleshooting for common failure patterns: sensor timeout (source data not ready), Spark OOM, driver crash, S3/MinIO access errors, iceberg-rest connectivity. |

Skills are markdown files loaded by `getSkill` — the agent loads the skill before starting any workflow (same as Data Architect).

## Canvas Components

All rendered via `@json-render/react` using `defineCatalog()` and `defineRegistry()`.

| Component | Purpose |
|---|---|
| `DagListCard` | Table of DAGs with status badge (active/paused), schedule interval, last run result (success/failed/running), sparkline of recent run outcomes |
| `DagRunTimeline` | Timeline/table of runs for a single DAG — logical date, state, start time, duration, action buttons |
| `TaskGraph` | Visual task dependency graph with per-task status coloring (green=success, red=failed, blue=running, gray=pending) |
| `TaskLogViewer` | Scrollable, syntax-highlighted log viewer for task instance stdout/stderr |
| `RunTriggerForm` | Form for selecting logical date + optional config overrides for manual triggers |
| `BackfillForm` | Date range picker, parallelism slider, strategy selector (sequential/parallel) |
| `StatusBadge` | Colored status pill — reuse pattern from data-architect, extended with run states (success, failed, running, queued, up_for_retry) |

### Canvas → Chat Actions

- `DagListCard` rows are clickable — sends `"Show details for <dag_id>"` to chat
- `DagRunTimeline` has a "Retry failed" button per run — sends `"Retry failed tasks in <run_id>"` to chat
- `RunTriggerForm` submit button — sends `"Trigger run for <dag_id> on <ds>"` to chat
- `BackfillForm` submit button — sends `"Backfill <dag_id> from <start> to <end>"` to chat

## Airflow API Integration

Airflow 3.x REST API. Tools hit `AIRFLOW_API_URL` (default: `http://localhost:8088` for local dev, `http://airflow-api-server.airflow:8080` in-cluster).

| Endpoint | Tool | Method |
|---|---|---|
| `/api/v2/dags` | `listDags` | GET |
| `/api/v2/dags/{dag_id}` | `getDagDetail` | GET |
| `/api/v2/dags/{dag_id}/dagRuns` | `listDagRuns` | GET |
| `/api/v2/dags/{dag_id}/dagRuns` | `triggerDagRun` | POST |
| `/api/v2/dags/{dag_id}/dagRuns/{run_id}/taskInstances` | `getTaskInstances` | GET |
| `/api/v2/dags/{dag_id}/dagRuns/{run_id}/taskInstances/{task_id}/logs` | `getTaskLogs` | GET |
| `/api/v2/dags/{dag_id}` | `toggleDagPause` | PATCH (`is_paused`) |
| `/api/v2/dags/{dag_id}/dagRuns/{run_id}/taskInstances/{task_id}` | `retryTask` | PATCH (clear state) |

### Filtering to Lattik-managed DAGs

All DAGs generated by `dag-generator.ts` carry a `lattik` tag. The `listDags` tool filters by this tag so the agent only shows DAGs it's responsible for — not `example_dag` or other unrelated DAGs.

### Cross-referencing with definitions DB

`getDagDetail` joins the Airflow API response with the `definitions` table (matched by `dag_id` → definition `name` + `kind`). This gives the user full context: the Lattik Table spec (column families, sources, schedule) alongside the Airflow operational state.

### Environment variable

Add to `apps/web/.env`:

```
AIRFLOW_API_URL=http://localhost:8088
```

## Agent Instructions (summary)

The system prompt follows the same structure as Data Architect:

1. **Role** — "You are the Pipeline Manager agent in Lattik Studio. You help users monitor, trigger, and troubleshoot Airflow DAGs for their Lattik Tables."
2. **Available skills** — Injected from `skills/index.ts`
3. **How to work** — Load skill first via `getSkill`, follow the skill doc. Canvas-first for status views.
4. **Canvas rendering** — On any monitoring request, first tool call after `getSkill` must be `renderDagOverview`. Don't ask clarifying questions — render the overview, let the user click into details.
5. **Confirmation for mutations** — `triggerDagRun`, `triggerBackfill`, `retryTask`, and `toggleDagPause` require explicit user confirmation before executing. Show what will happen, ask "proceed?", then act.
6. **Off-topic / handback** — Same protocol as Data Architect.
7. **Task completion** — Same protocol as Data Architect.

## Implementation Phases

### Phase 1 — Read-only monitoring

Scope: Users can view DAG status, drill into runs, and read task logs. No mutations.

Files:
- `agent.ts`, `register.ts` — agent definition and registration
- `lib/airflow-client.ts` — typed Airflow REST client
- `tools/get-skill.ts`, `tools/list-dags.ts`, `tools/get-dag-detail.ts`, `tools/list-dag-runs.ts`, `tools/get-task-instances.ts`, `tools/get-task-logs.ts`, `tools/read-canvas-state.ts`
- `canvas/` — `DagListCard`, `DagRunTimeline`, `TaskGraph`, `TaskLogViewer`, `StatusBadge`
- `skills/monitoring-dags.md`
- `extensions/index.ts` — add import for side-effect registration

### Phase 2 — Actions

Scope: Trigger runs, backfills, retry failed tasks, pause/unpause DAGs.

Files:
- `tools/trigger-dag-run.ts`, `tools/trigger-backfill.ts`, `tools/retry-task.ts`, `tools/toggle-dag-pause.ts`
- `canvas/` — `RunTriggerForm`, `BackfillForm`
- `skills/triggering-runs.md`
- Update `agent.ts` instructions for confirmation-before-mutation protocol

### Phase 3 — Troubleshooting

Scope: Guided troubleshooting with enriched error context.

Files:
- `skills/troubleshooting-failures.md` — common failure patterns (sensor timeout, Spark OOM, driver crash, S3 access denied, iceberg-rest unreachable), with diagnostic steps for each
- Optionally enrich `getTaskLogs` to parse known error patterns and surface structured diagnostics

## Logger Table monitoring (planned)

The agent's identity covers Logger Table health, but the tools, skills, and canvas components for it are not yet implemented. This section is a placeholder for the design — to be filled in as a follow-up to the framing change.

Candidate signals to surface for a single Logger Table:

- **Kafka topic state** — partitions, current end offsets, consumer-group lag for the logger-writer.
- **Logger-writer pod state** — pod phase, restart count, recent log tail.
- **Iceberg commit health** — last commit timestamp, rows committed since some window, file count / small-file pressure.
- **Schema state** — registered Protobuf schema version in Schema Registry, last schema change.
- **Throughput** — events/sec produced (Kafka) and rows/sec committed (Iceberg) over a recent window.

Open questions (to resolve before building):

- Single "ecosystem overview" canvas (Logger Tables + DAGs together) vs. separate views?
- Should drill-down into a Logger Table show recent events (sampled rows from Iceberg) or only operational signals?
- Where do throughput metrics come from in local dev — JMX scrape of Kafka, or a derived count from Iceberg snapshots?

## Key Design Decisions

1. **Airflow client as extension-scoped lib** — A typed wrapper around the Airflow REST API in `lib/airflow-client.ts`. Handles connection config, error handling, response types. Scoped to this extension, not a shared utility.

2. **Filter by `lattik` tag** — Only show DAGs generated by the platform. Prevents confusion with unrelated DAGs in the same Airflow instance.

3. **Cross-reference with definitions DB** — DAG detail includes the Lattik Table spec so users see pipeline config alongside operational state without switching agents.

4. **Canvas-first for status, chat for actions** — Status views (DAG list, run history, task graph) render on canvas. Mutations (trigger, retry, pause) are conversational with explicit confirmation.

5. **No DAG editing** — Separation of concerns. The Pipeline Manager observes and operates; the Data Architect designs and defines.

6. **Backfill as first-class workflow** — Backfills are common and error-prone. A dedicated skill walks users through date range selection, parallelism tuning, and progress monitoring.

7. **Phased implementation** — Read-only first (Phase 1), then actions (Phase 2), then troubleshooting intelligence (Phase 3). Each phase is independently useful.
